# Steam PC 多人联机卡牌游戏网络方案开发指导

> 项目背景：  
> 游戏类型类似《杀戮尖塔》的多人合作卡牌 Roguelike，包含房主开房间、玩家加入房间、职业选择、回合制战斗、卡牌/技能/道具结算、网格移动、Buff/Debuff、随机奖励、地图路线选择等系统。
>
> 当前目标：  
> **不自建专用服务器**，由房主电脑承担服务器功能；开发阶段优先支持局域网验证，允许公网 IP 主机作为房主；正式发布目标为 **Steam PC**。

---

## 1. 最终推荐结论

推荐采用：

```text
Unity + Mirror + Host 模式
开发阶段：KCP / LAN / IP 直连
Steam 阶段：Mirror + Steam Transport / FizzySteamworks
同步方式：房主权威 + 操作请求 + 事件广播 + 状态快照
```

核心原则：

```text
房主客户端 = Host = Server + Client
其他玩家 = Client
房主机器负责战斗结算
普通客户端只发送操作请求
所有客户端根据服务器广播的事件播放动画和刷新 UI
```

---

## 2. 为什么不选专用服务器方案

Nakama、Colyseus、专用 Dedicated Server 都适合长期运营型多人游戏，但它们需要你维护服务器或云资源。

你当前明确希望：

```text
不自己提供服务器
不在乎房主作弊
房主电脑承担服务器功能
Steam PC 发布
```

因此更适合：

```text
Host / Listen Server 模式
```

也就是房主既是玩家，也是服务器。

---

## 3. 技术选型

### 首选：Mirror

推荐原因：

1. 开源免费，Unity 生态成熟。
2. 支持 Host 模式。
3. 可以继续使用 C# 编写战斗逻辑。
4. 适合你的回合制低频同步。
5. 后续可以通过 Steam Transport / FizzySteamworks 接入 Steam 网络。
6. 不需要把每张卡牌、每个 Buff 都做成 NetworkObject。

Mirror 在你的项目里应该被当成：

```text
消息传输层 + 房间连接层
```

而不是：

```text
大量同步 GameObject 的实时动作框架
```

---

### 开发阶段 Transport

开发阶段建议使用：

```text
Mirror + KCP Transport
```

适合：

```text
局域网联机
公网 IP + 端口开放直连
基础 Host/Client 验证
```

典型端口：

```text
7777
```

---

### Steam 发布阶段 Transport

Steam 阶段建议使用：

```text
Mirror + FizzySteamworks
```

或者其他兼容 Mirror 的 Steam Transport。

目的：

```text
使用 Steam P2P / Steam Relay 能力
降低 NAT 穿透问题
支持 Steam 好友邀请 / Lobby
避免玩家手动配置端口映射
```

## 4. 联网模式设计

### 4.1 开发阶段：局域网 / 公网 IP 直连

结构：

```text
房主电脑 StartHost()
其他玩家输入房主 IP 加入
```

示意：

```text
Client B  ─┐
Client C  ─┼──> Host A
Client D  ─┘
```

适合验证：

```text
创建房间
加入房间
同步玩家列表
选择职业
开始战斗
出牌
移动
回合结束
敌人行动
战斗胜负
奖励选择
```

不适合验证：

```text
普通玩家远程互联网稳定连接
无公网 IP 房主联机
Steam 好友邀请
断线重连完整体验
```

---

### 4.2 Steam 阶段：Steam P2P / Relay

结构：

```text
Client B  ─┐
Client C  ─┼── Steam P2P / Relay ── Host A
Client D  ─┘
```

注意：

```text
房主仍然是服务器
Steam 只负责连接和转发
你不需要自建战斗服务器
```

这正好符合你的目标：

```text
不维护服务器
房主负责结算
远程玩家能加入
```

## 5. NAT / 内网穿透影响

### 5.1 成员没有公网 IP

通常影响较小。

原因：

```text
成员是主动向外连接
大多数 NAT 允许内网机器主动访问外部
```

只要房主可以被访问，成员通常可以加入。

---

### 5.2 房主没有公网 IP

影响很大。

如果使用普通 IP 直连：

```text
房主没有公网 IP
房主在 CGNAT / 校园网 / 公司网 / 移动热点后面
路由器没有端口映射
防火墙阻挡端口
```

其他玩家大概率连不进来。

---

### 5.3 开发阶段解决方式

开发阶段可以接受以下限制：

```text
优先局域网测试
或使用有公网 IP 的电脑作为房主
或手动配置端口映射
```

开发阶段不建议过早处理复杂 NAT 穿透。

---

### 5.4 Steam 阶段解决方式

Steam 发布阶段建议通过：

```text
Steam P2P / Steam Relay
```

降低 NAT 问题。

---

## 6. 同步架构

你的游戏不应该做帧同步。

应该采用：

```text
房主权威状态同步
```

更具体是：

```text
ActionRequest -> Host Validate -> Host Resolve -> BattleEvents -> Snapshot
```

---

### 6.1 客户端发送操作请求

普通客户端只发送“我要做什么”。

例如：

```json
{
  "type": "PlayCard",
  "playerId": "P1",
  "cardInstanceId": "card_102",
  "targetId": "enemy_01",
  "clientActionId": 58
}
```

移动：

```json
{
  "type": "Move",
  "playerId": "P1",
  "from": [2, 3],
  "to": [2, 5],
  "clientActionId": 59
}
```

结束回合：

```json
{
  "type": "EndTurn",
  "playerId": "P1",
  "clientActionId": 60
}
```

---

### 6.2 房主服务器验证请求

Host 检查：

```text
是否轮到玩家阵营
玩家是否存活
卡牌是否在手牌中
能量是否足够
体力是否足够
目标是否合法
距离是否合法
当前是否被眩晕 / 沉默 / 禁止移动
操作是否重复提交
```

---

### 6.3 房主服务器结算结果

Host 执行：

```text
消耗能量
消耗体力
打出卡牌
造成伤害
添加 Buff
移动单位
触发碰撞
触发死亡
触发抽牌
移动卡牌到弃牌堆 / 消耗堆
判断胜负
生成奖励
```

---

### 6.4 广播事件流

Host 广播 BattleEvents：

```json
{
  "serverActionId": 301,
  "events": [
    {
      "type": "CardPlayed",
      "playerId": "P1",
      "cardInstanceId": "card_102"
    },
    {
      "type": "EnergyChanged",
      "playerId": "P1",
      "from": 3,
      "to": 1
    },
    {
      "type": "Damage",
      "sourceId": "P1",
      "targetId": "enemy_01",
      "amount": 12
    },
    {
      "type": "BuffAdded",
      "targetId": "enemy_01",
      "buffId": "Vulnerable",
      "stack": 2
    }
  ]
}
```

客户端根据事件播放动画。

---

### 6.5 同步状态快照

每次关键操作后或每回合结束时同步 Snapshot：

```json
{
  "turn": 4,
  "currentSide": "Player",
  "players": [],
  "enemies": [],
  "hands": {},
  "drawPileCounts": {},
  "discardPileCounts": {},
  "mapUnits": []
}
```

Snapshot 用来纠正客户端显示状态，避免表现层因为动画或丢包产生偏差。

## 7. Mirror 实现结构建议

### 7.1 不建议的做法

不要这样设计：

```text
每张卡牌 = NetworkObject
每个 Buff = NetworkObject
每个牌堆 = NetworkList
每个 UI 手牌对象 = NetworkObject
```

这样会让卡牌游戏变得非常复杂。

---

### 7.2 推荐做法

推荐：

```text
一个 BattleNetworkController 负责网络消息
一个 BattleState 负责真实战斗状态
一个 BattleResolver 负责结算规则
一个 BattleEventPlayer 负责客户端动画播放
```

结构：

```text
NetworkManager
 ├── LobbyNetworkController
 ├── BattleNetworkController
 ├── BattleState
 ├── ActionQueue
 ├── ActionValidator
 ├── BattleResolver
 ├── BattleEventBroadcaster
 └── SnapshotBuilder
```

---

## 8. 关键类设计

### 8.1 BattleNetworkController

负责接收客户端请求，并广播结果。

```csharp
public class BattleNetworkController : NetworkBehaviour
{
    [Command]
    public void CmdPlayCard(string requestJson)
    {
        if (!isServer) return;

        var request = JsonUtility.FromJson<PlayCardRequest>(requestJson);
        var result = BattleServer.Instance.HandlePlayCard(request, connectionToClient);

        RpcApplyBattleEvents(result.EventsJson);
        RpcApplySnapshot(result.SnapshotJson);
    }

    [ClientRpc]
    private void RpcApplyBattleEvents(string eventsJson)
    {
        BattleClient.Instance.PlayEvents(eventsJson);
    }

    [ClientRpc]
    private void RpcApplySnapshot(string snapshotJson)
    {
        BattleClient.Instance.ApplySnapshot(snapshotJson);
    }
}
```

---

### 8.2 BattleServer

只在 Host/Server 端运行。

```csharp
public class BattleServer
{
    public BattleState State;
    public ActionQueue Queue;
    public ActionValidator Validator;
    public BattleResolver Resolver;

    public BattleActionResult HandlePlayCard(
        PlayCardRequest request,
        NetworkConnectionToClient sender)
    {
        var playerId = GetPlayerId(sender);

        if (!Validator.ValidatePlayCard(State, playerId, request, out var error))
        {
            return BattleActionResult.Failed(error);
        }

        var events = Resolver.ResolvePlayCard(State, playerId, request);
        var snapshot = SnapshotBuilder.Build(State);

        return BattleActionResult.Success(events, snapshot);
    }
}
```

---

### 8.3 BattleClient

只负责表现。

```csharp
public class BattleClient
{
    public void PlayEvents(string eventsJson)
    {
        // 播放出牌、伤害、移动、Buff 等动画
    }

    public void ApplySnapshot(string snapshotJson)
    {
        // 用服务器快照刷新 UI 和场上状态
    }
}
```

## 9. 房主权限设计

房主不是普通意义上的“管理员服务器”，但可以拥有房间权限。

房主可以做：

```text
创建房间
邀请玩家
踢出玩家
开始游戏
选择大地图路线
选择进入下一关
返回大厅
解散房间
```

房主不应该通过客户端 UI 直接改战斗结果。

即使不在乎房主作弊，也应该保持架构清晰：

```text
房主客户端 UI 发送请求
房主本机 Server 逻辑结算
房主客户端表现层接收结果
```

也就是说，房主玩家也要走同一套 ActionRequest 流程。

## 10. 开发阶段里程碑

### Milestone 1：Mirror 最小联机验证

目标：

```text
房主 StartHost
成员 StartClient
成员加入房间
同步玩家列表
房主开始游戏
```

完成标准：

```text
两台电脑在局域网内可以加入同一个房间
所有玩家能看到玩家列表
房主可以点击开始
所有玩家切换到战斗场景
```

---

### Milestone 2：职业选择与准备状态

目标：

```text
玩家选择职业
同步职业选择
所有玩家 Ready
房主开始战斗
```

完成标准：

```text
玩家职业在所有客户端显示一致
只有房主能开始
未 Ready 时不能开始
```

---

### Milestone 3：战斗状态快照

目标：

```text
Host 初始化 BattleState
同步玩家属性
同步手牌
同步敌人
同步地图单位位置
```

完成标准：

```text
所有客户端进入战斗后显示一致
每个玩家只能看到自己的手牌具体内容
其他玩家只显示手牌数量
```

---

### Milestone 4：出牌请求与结算

目标：

```text
客户端提交 CmdPlayCard
Host 验证合法性
Host 结算伤害 / Buff / 资源
广播 BattleEvents
同步 Snapshot
```

完成标准：

```text
普通攻击牌可以完整打出
能量正确扣除
敌人生命正确变化
卡牌进入弃牌堆
所有客户端显示一致
```

---

### Milestone 5：移动请求与网格冲突

目标：

```text
客户端提交 CmdMove
Host 验证体力和路径
Host 结算移动
处理单位碰撞 / 挤出 / 阻挡
广播移动事件
```

完成标准：

```text
移动消耗体力
不能走进非法格子
被动移动不消耗体力
冲突规则由 Host 统一处理
```

---

### Milestone 6：回合流程

目标：

```text
玩家阵营回合
敌人阵营回合
回合开始 Buff
资源恢复
抽牌
回合结束弃牌
敌人 AI
胜负判断
```

完成标准：

```text
完整战斗循环可以运行
多人玩家可以在玩家回合内按提交顺序操作
敌人行动由 Host 统一执行
```

---

### Milestone 7：断线和重连基础处理

目标：

```text
玩家断线
保留角色状态
玩家重连后同步 Snapshot
```

完成标准：

```text
短时间断线重连后可以继续游戏
重连玩家拿到完整当前状态
如果房主断线，房间结束
```

注意：

```text
Host 模式下房主断线通常无法无缝迁移
初期可以接受房主断线 = 房间结束
```

---

### Milestone 8：Steam 联机验证

目标：

```text
接入 Steam Transport
创建 Steam Lobby
好友加入
通过 Steam P2P / Relay 连接
```

完成标准：

```text
不同网络环境下好友可以加入房间
不需要手动输入公网 IP
不需要玩家手动端口映射
```

## 12. Host 断线问题

Host 模式最大缺陷：

```text
房主退出 / 崩溃 / 断网 = 房间结束
```

可选方案：

### 初期方案

```text
房主断线，所有玩家返回主菜单
提示：房主已离开，游戏结束
```

这是最简单、最现实的方案。

---

### 进阶方案：保存战斗进度

房主定期保存：

```text
BattleState
地图进度
玩家状态
卡组状态
奖励状态
```

如果房主重新开房，可以从存档恢复。

但这通常只适合固定好友队伍。

---

### 不建议初期做：Host Migration

Host Migration 指房主离开后，另一个玩家接管服务器。

不建议初期实现，原因：

```text
复杂度高
状态迁移难
Steam Lobby 所有权迁移也要额外处理
卡牌/随机数/战斗状态容易出错
```

---

## 13. 随机数规则

所有随机数必须由 Host 产生。

包括：

```text
抽牌顺序
奖励卡牌
地图节点
怪物组合
敌人意图
事件选项
商店商品
副职业奖励
```

不要让客户端自己随机。

推荐：

```text
BattleRandomService
```

并保存：

```text
seed
randomCallIndex
```

这样方便调试和回放。

## 19. 常见风险与处理

### 风险 1：房主无公网 IP，开发阶段连不上

处理：

```text
开发阶段先用局域网
公网测试使用有公网 IP 的机器
Steam 阶段再解决 NAT
```

---

### 风险 2：把太多对象做成 NetworkObject

处理：

```text
卡牌、Buff、牌堆、敌人状态优先走 BattleState 数据同步
少用 NetworkObject
多用 Command / Rpc / 自定义消息
```

---

### 风险 3：客户端和服务器逻辑重复

处理：

```text
真实结算只在 Host Server 执行
客户端只做预测高亮和表现
客户端预测不改变真实状态
```

---

### 风险 4：动画和状态不同步

处理：

```text
服务器先结算
广播 BattleEvents
客户端按事件顺序播放动画
动画结束后应用 Snapshot 或校正显示
```

---

### 风险 5：多人同时操作冲突

处理：

```text
Host 维护 ActionQueue
按服务器收到请求的顺序结算
同一时刻只处理一个修改状态的 Action
```
