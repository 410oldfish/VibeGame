# 《VibeGame》遭遇与原型战斗设计

> 文档状态：草案 v0.2
> 关联总案：[`../游戏粗略策划案.md`](../游戏粗略策划案.md)

## 1. 文档定位

本文件是遭遇系统的内容主文档，负责说明战斗内容如何被编排、教学、检验和拔高。

本文件负责：

- 普通战、精英战和首领战的内容职责。
- 遭遇编排的学习顺序。
- 首个可玩原型的内容范围。

本文件不负责：

- 战斗底层回合规则。
- 地形系统词典。
- 敌方牌组的通用系统规则。

## 2. 设计目的

- 让遭遇内容承担教学、检验和拔高三种不同职责。
- 让普通战、精英战和首领战分别对应不同层级的系统理解要求。
- 让首个可玩原型在有限范围内验证六边形移动、敌方意图与地形改装三大核心。

## 3. 系统边界

| 类型 | 主要内容 |
| --- | --- |
| 上游依赖 | 冒险节点、战斗规则、敌人模板、地形模板、奖励节奏 |
| 系统产出 | 具体战斗内容组合、教学顺序、章节内容强度曲线、原型编排方案 |
| 下游影响 | 玩家学习路径、章节体验密度、测试结论质量 |

## 4. 系统规则

### 4.1 遭遇职责划分

| 遭遇类型 | 主要职责 |
| --- | --- |
| 普通战斗 | 教会玩家一种清晰交互 |
| 精英战斗 | 检验玩家当前构筑方向是否成立 |
| 首领战斗 | 总结章节主题并要求玩家主动改写战场 |

### 4.2 不同遭遇的设计重点

- 普通战斗：单场只强调一到两个关键词，例如“击退 + 墙体”或“水域 + 感电”。
- 精英战斗：必须有能够循环出现的危险点，并至少提供两种破解方式。
- 首领战斗：应通过阶段性牌组和战场改写形成压迫感，而不是只靠数值碾压。

## 5. 核心机制

### 5.1 遭遇是系统教学器

遭遇系统的核心不是“把敌人堆在地图上”，而是让玩家按顺序学会移动、站位、地形、敌方意图和构筑收益。

### 5.2 原型范围必须收窄

首个原型优先回答三个问题：

1. 六边形移动与出牌结合后，每回合的位置抉择是否有趣且清晰？
2. 敌人公开意图并使用牌堆循环后，玩家是否感到在“应对一手牌”而非机械躲技能？
3. 少量可转化地形是否能产生值得围绕构筑的策略，而不会造成认知负担？

## 6. 子系统

当前遭遇系统下暂未拆出更多独立子系统文档。

后续可继续拆分为：

- 普通战模板。
- 精英与首领模板。
- 原型场景编排。

## 7. 当前原型范围

| 模块 | 原型内容 |
| --- | --- |
| 战场 | `1` 张中等尺寸六边形竞技场地图 |
| 玩家 | 优先支持战士、骑士、德鲁伊各 `1` 套精简初始牌组 |
| 敌人 | `2` 种普通敌人 + `1` 种精英，各自有小型牌组和公开意图 |
| 地形 | 普通地面、墙体、深坑、水域；临时覆盖先做火焰与电场 |
| 改装 | 生成墙体、点燃地面、水域通电、破坏墙体共 `4` 类动作 |
| 状态 | 力量、格挡、燃烧、感电、束缚/稳固等直接服务于原型的状态 |
| 胜负 | 一场完整战斗及战斗奖励三选一卡牌 |
| 多人 | 若联机验证同步进行，先完成 `2` 名玩家合作与房主权威结算 |

原型阶段暂不纳入：

- 完整三章内容和全部事件/商店内容。
- 复杂地形元素连锁、动态天气或大量可交互设施。
- 大量敌方牌库操纵牌。
- Host 迁移、公开匹配等发布期网络需求。

## 9. 约束与例外

- 本文只定义内容编排，不是战斗底层规则源。
- 若未来形成章节级敌人库、区域主题库或首领脚本库，应拆出独立内容文档。
- 原型阶段必须主动控制变量数量，避免系统结论失真。
