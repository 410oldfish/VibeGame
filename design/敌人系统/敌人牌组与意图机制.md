# 《VibeGame》敌人牌组与意图机制

> 文档状态：草案 v0.2
> 关联总案：[`../游戏粗略策划案.md`](../游戏粗略策划案.md)

## 1. 文档定位

本文件是敌人系统的主文档，负责定义敌方牌组、公开意图、执行流程以及玩家如何与敌方“打牌”。

本文件负责：

- 敌方牌区与信息公开层级。
- 敌方意图的生成和执行框架。
- 玩家干涉敌方牌局的方式。
- 防止敌人系统被完全锁死的约束规则。

本文件不负责：

- 敌人在六边形空间中的移动合法性。
- 具体地形词典与地形效果。
- 状态词条表本身。

## 2. 设计目的

敌人的牌组机制需要同时满足：

- 可预判：玩家能看到足够信息，从而通过移动和构筑做出决策。
- 有变化：相同敌人在不同洗牌与地形下不会完全按脚本重复。
- 可干涉：玩家不只承受攻击，还能影响敌人下一轮会如何出牌。
- 可读性优先：不要求玩家记住敌人整套牌库才能做出合理选择。

## 3. 系统边界

| 类型 | 主要内容 |
| --- | --- |
| 上游依赖 | 战场位置、敌人模板、敌方牌组、状态效果、地形约束、遭遇编排 |
| 系统产出 | 敌方公开意图、敌方执行结果、敌方牌区变化、敌人压迫节奏 |
| 下游影响 | 玩家移动与出牌决策、打断与诱导选择、遭遇难度表现 |

## 4. 系统规则

### 4.1 敌方牌区

建议每个主要敌人拥有以下数据：

| 区域/信息 | 对玩家是否公开 | 说明 |
| --- | --- | --- |
| 敌人牌库构成摘要 | 部分公开 | 遭遇开始时显示关键词，例如“冲撞 / 燃烧 / 防守” |
| 抽牌堆 | 数量公开，顺序隐藏 | 首领可通过阶段改变牌库 |
| 当前意图手牌 | 公开 | 展示敌方下一敌人回合准备打出的牌及目标范围 |
| 弃牌堆 | 可查看 | 玩家可判断危险牌多久可能再次出现 |
| 消耗/封印区 | 公开 | 记录被永久处理或阶段性禁用的敌方牌 |

普通杂兵可共享简化牌组或仅拥有 `3` 至 `5` 张个人牌；精英和首领应拥有更鲜明的牌组循环。

### 4.2 敌方执行流程

```text
敌方意图生成
  -> 玩家回合读取威胁并做出响应
  -> 敌方执行公开意图
  -> 未执行或被打断的牌进入对应牌区
  -> 敌方生成下一轮可见意图
```

建议在战斗开始时先公开第一轮敌方意图，再给予玩家先手，以保障第一回合决策质量。

### 4.3 玩家与敌人“打牌”的交互类型

| 交互方式 | 规则概念 | 示例 |
| --- | --- | --- |
| 躲避 | 离开意图攻击覆盖区 | 敌人展示直线突进，玩家移出直线 |
| 阻挡 | 通过防御或地形使牌收益降低 | 骑士升起掩体挡住射线 |
| 诱导 | 改变敌人位置或可选目标 | 嘲讽后将重击引向高格挡角色 |
| 打断 | 满足条件后取消公开意图 | 眩晕敌人，使其蓄力攻击进入弃牌堆 |
| 污染牌组 | 向敌方牌堆加入弱化牌/状态牌 | 向敌人弃牌堆塞入“卡壳” |
| 窃取/复制 | 获得敌人牌的临时友方版本 | 精英掉落或职业特色机制 |
| 强迫出牌 | 令敌人提前在不佳位置执行牌 | 诱发冲撞撞墙或提前引爆区域 |

## 5. 核心机制

### 5.1 敌方公开信息是战斗决策的一部分

敌人系统的关键不在于“敌人会不会攻击”，而在于玩家能否在敌人行动前看懂威胁，并通过移动、构筑、地形或状态去改变结果。

### 5.2 敌人也拥有牌局节奏

同一个敌人不应只靠固定脚本循环。其牌组应通过抽牌、弃牌、阶段变化和地形差异，形成可读但不僵化的压力节奏。

### 5.3 首领需要最低行动保障

干涉敌方牌组的卡牌应控制数量与费用，防止玩家彻底锁死首领；首领可拥有抗性、阶段解锁牌或每轮最低行动保障。

## 6. 子系统

当前敌人系统下暂未拆出独立的敌人模板文档。

后续可继续拆分为：

- 普通敌人模板。
- 精英与首领模板。
- 敌方抗干涉规则。

## 8. 约束与例外

- 本文只定义敌方牌局框架，不写具体场景编排与战场地形表。
- 信息公开应足够支撑规划，但不能让所有风险完全静态化。
- 若未来加入群体 AI、共享敌方资源或反制逻辑，需要另建子文档而不是塞进遭遇文档。
